# kops 术语说明

## RPS（Requests Per Second）

**每秒请求数**。衡量服务接收流量的核心指标，通过 Prometheus 从 Traefik 采集。

- 高 RPS 表示服务承载大量请求
- RPS 为 0 或极低（<0.001）表示服务几乎没有流量，可能处于空闲状态

在健康检查中，RPS < 0.001 的服务会被标记为 **空闲（Idle）**。

---

## 流量密度（Traffic Density）

**RPS / CPU 核心数**，即每消耗 1 个 CPU 核心能处理多少请求。

```
流量密度 = AvgRPS ÷ (CPUUsageAvg / 1000)
```

它衡量的是 **CPU 资源的产出效率**——花同样的 CPU，谁扛的流量更多。

| 评级 | 阈值 | 含义 |
|------|------|------|
| S (极高) | > 2000 RPS/Core | 资源利用效率极高 |
| A (正常) | 500 ~ 2000 RPS/Core | 流量与 CPU 配比合理 |
| B (低效) | 50 ~ 500 RPS/Core | CPU 利用偏低，建议缩容 |
| C (极低/空转) | ≤ 50 RPS/Core | 几乎空转，资源严重浪费 |

> 如果大量服务处于 C 级，说明集群普遍过度配置（CPU request 远超实际需求）。

---

## P99 延迟（P99 Latency）

**99 分位延迟**，即在所有请求中，**99% 的请求**在这个时间内完成。

例如 P99 = 200ms，意味着：

- 99% 的请求在 200ms 内返回
- 只有 1% 的请求超过 200ms

对比 P50（中位数）和 P95：

| 指标 | 含义 |
|------|------|
| P50 | 一半请求的耗时 |
| P95 | 95% 请求的耗时 |
| P99 | 99% 请求的耗时，更能反映「长尾」慢请求 |

P99 比平均值更能反映真实用户体验——平均值会被少量极快请求拉低，而 P99 暴露的是那 1% 用户实际感受到的延迟。

---

## 5xx / 4xx 错误率

通过 Traefik 采集的 HTTP 状态码比例：

- **5xx 错误率**：服务端错误占比（500/502/503/504），高 5xx 说明服务自身有问题
- **4xx 错误率**：客户端错误占比（400/401/403/404），高 4xx 可能是接口变更或权限问题

公式：`ErrorRate = 5xx(或4xx)请求数 / 总请求数`

---

## 资源黑洞（Resource Black Hole）

**高成本、近零利用**的服务。满足两个条件：

1. 当前月成本 > 配置的阈值（默认通过 `Governance.BlackHoleCostThreshold` 控制）
2. 实际资源消耗 / 分配资源 < 10%（即浪费比例 > 90%）

这类服务是最优先的缩容/下线目标。

---

## 无效支出（Invalid Spend）

因健康问题（高错误率、高延迟）导致的「花了钱但没产生价值」的支出。计算逻辑：服务不健康时，其成本按比例计入无效支出。
